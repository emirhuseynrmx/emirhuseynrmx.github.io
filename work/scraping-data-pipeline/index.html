Problem neydi? Neden bu yaklaşım? What was the problem? Why this approach?

Müşteri sorunuClient problem E-ticaret şirketi, 3 rakibin 500 ürününü takip ediyordu — elle, Google Sheets'e kopyala-yapıştır. Haftada 6 saat. Rakip site tasarımı değişince script çöküyor, yeniden kod yazılıyor.E-commerce company tracking 500 products across 3 competitors — manually, copy-paste to Google Sheets. 6 hours/week. When the competitor's site redesigns, the script breaks, code must be rewritten.
selectolax ← BeautifulSoup4 Python'da yazılmış, yavaş. selectolax C tabanlı lexbor engine — aynı HTML'i parse etmek için selectolax 0.08s, BS4 0.82s (10x). 1000 sayfada bu fark 12 dakika vs 800 saniye. Büyük scrape'lerde seçim değişmiyor.BeautifulSoup4 is written in Python, slow. selectolax uses C-based lexbor engine — parsing the same HTML: selectolax 0.08s, BS4 0.82s (10x). At 1000 pages that's 12 minutes vs 800 seconds. For large-scale scraping, no contest.
async httpx ← requests senkron: 100 sayfa × 0.5s/sayfa = 50 saniye bekleme. httpx + asyncio: 100 istek aynı anda gönderilir, toplam süre en yavaş isteğin süresi ≈ 1.8s. Concurrency=12 olarak ayarlandı (hedef sunucuyu zorlamamak için).requests is synchronous: 100 pages × 0.5s/page = 50 seconds wait. httpx + asyncio: 100 requests sent simultaneously, total time = slowest request ≈ 1.8s. Concurrency set to 12 (to avoid overwhelming the target server).
Config-driven ← Rakip site CSS selector'ı değişirse kodu açmak zorunda kalma. targets.json'da url, selector, field mapping güncelle, çalıştır. Müşteri bu güncellemeyi kendisi yapabiliyor — teknik bilgi gerekmez.If a competitor's CSS selector changes, no need to open the code. Update url, selector, field mapping in targets.json and run. Client can make this update themselves — no technical knowledge required.

Adım adım senaryo: 3 site, 2363 kayıt Step-by-step scenario: 3 sites, 2363 records

Gece 02:00'de cron job tetikleniyor. Müşteri uyurken sistem 3 rakibin tüm ürün sayfalarını tarıyor. Cron job triggers at 02:00. While the client sleeps, the system scans all product pages of 3 competitors.

Sıralı yapılsaydı ≈48 saniye. example.com → 847 items (1.2s). shop.example.io → 1204 items (2.8s). news.example.net → timeout on first try, success on retry=2 → 312 items. Total: 2363 records, 4.2 seconds. Sequential would take ≈48 seconds.

02

Pydantic doğrulama: 12 kayıt reddedildiPydantic validation: 12 records rejected

2363 kayıttan 12'si Pydantic modelden geçemedi: 7'sinde fiyat alanı boş (HTML'de selector bulunamamış), 3'ünde fiyat string ama float bekleniyor ("N/A"), 2'si eksik ürün başlığı. Bunlar hata log'una yazıldı, pipeline durdurmadı. 12 of 2363 records failed Pydantic validation: 7 with empty price field (selector not found in HTML), 3 with price as string but float expected ("N/A"), 2 with missing product title. These were written to error log, pipeline didn't stop.

03

Pandera + export: 2351 temiz kayıtPandera + export: 2351 clean records

2351 geçerli kayıt Pandera schema'dan geçirildi (fiyat ≥ 0, ürün_adı boş olamaz, kaynak_url valide URL). Schema geçildi → Parquet formatında diske yazıldı. Sabah müşteri açtığında güncel rakip fiyat tablosu hazır. 2351 valid records passed Pandera schema (price ≥ 0, product_name cannot be empty, source_url must be valid URL). Schema passed → written to disk as Parquet. When the client opens it in the morning, competitor price table is current.
