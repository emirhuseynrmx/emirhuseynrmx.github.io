Problem tipi: async I/O + durum makinesiProblem type: async I/O + state machine

Bu bir ML problemi değil. Buradaki teknik zorluk: çok sayıda eşzamanlı kullanıcı konuşmasını verimli yönetmek ve her kullanıcı için konuşma durumunu (hangi aşamada?) tutmak. Asyncio + FSM (Finite State Machine) bu iki sorunu birlikte çözüyor. Not an ML problem. The technical challenge here: efficiently managing many concurrent user conversations and maintaining the conversation state (which step?) per user. Asyncio + FSM (Finite State Machine) solve these two problems together.

Neden aiogram 3.x? python-telegram-bot değil? Why aiogram 3.x? Not python-telegram-bot?

python-telegram-bot (sync) Senkron thread-based. Her kullanıcı konuşması = 1 thread. 50 eşzamanlı kullanıcıda 50 thread, hepsi network I/O'da bloke bekliyor. Thread overhead yüksek, bellek tüketimi lineer artıyor. FSM için harici state yönetimi gerekiyor (Redis veya dict hack).Synchronous thread-based. Each user conversation = 1 thread. At 50 concurrent users, 50 threads all blocking on network I/O. High thread overhead, memory grows linearly. External state management needed for FSM (Redis or dict hack).
aiogram 3.x Native asyncio. Tek event loop, 1000+ eşzamanlı konuşma — tek thread. FSM built-in: MemoryStorage, RedisStorage veya custom. State WAITING_NAME → WAITING_EMAIL → WAITING_BUDGET → COMPLETE doğrudan framework içinde tanımlanıyor. Dependency injection ile test yazımı kolaylaşıyor.Native asyncio. Single event loop, 1000+ concurrent conversations — single thread. Built-in FSM: MemoryStorage, RedisStorage or custom. States WAITING_NAME → WAITING_EMAIL → WAITING_BUDGET → COMPLETE defined directly in the framework. Dependency injection simplifies testing.
Pydantic v2 Lead verisi kayıt öncesi validate ediliyor. E-posta email-validator ile doğrulanıyor. Bütçe alanı sayısal zorunlu. Eksik alan → bot kullanıcıyı tekrar soruyor. Validasyonsuz hatalı lead CSV'e yazılırdı.Lead data validated before saving. Email validated with email-validator. Budget field must be numeric. Missing field → bot re-prompts the user. Without validation, malformed leads would write to CSV silently.

Adım adım senaryo: lead yakalama akışı Step-by-step scenario: lead capture flow

01

Kullanıcı /faq pricing yazarUser types /faq pricing

FAQ_RESPONSES dict'ine "pricing" key ile bak. Anında yanıt döndür. Türkçe içerik: Starter 1.500-3.000 EUR, Growth 3.000-8.000 EUR, Enterprise 8.000+. Yanıt süresi: <50ms. Hiçbir konuşma state'i tüketilmedi. Look up "pricing" key in FAQ_RESPONSES dict. Return answer instantly. Content: Starter 1,500-3,000 EUR, Growth 3,000-8,000 EUR, Enterprise 8,000+. Response time: <50ms. No conversation state consumed.

02

Kullanıcı /lead yazarUser types /lead

FSM WAITING_NAME state'ine girer. "Adınızı girin:" sorar. Kullanıcı "Ahmet Yılmaz" yazar. State WAITING_EMAIL'e geçer. E-posta gelir, email-validator ile doğrular. Hatalıysa tekrar sorar. Bütçe gelir, float parse edilir. Başarılıysa LeadCapture Pydantic modeline dönüştürülür, leads.csv'ye eklenir. FSM enters WAITING_NAME state. Asks "Enter your name:". User types "Ahmet Yilmaz". State moves to WAITING_EMAIL. Email received, validated with email-validator. If invalid, re-prompts. Budget received, parsed as float. On success, converted to LeadCapture Pydantic model, appended to leads.csv.

03

Admin /export_leads yazarAdmin types /export_leads

Admin kapısı kontrol edilir: gönderen settings.admin_ids listesinde mi? Değilse reddedilir. leads_csv_path'teki CSV okunur, Telegram'dan dosya olarak gönderilir. İçerik: isim, e-posta, bütçe, timestamp. FAQ yanıtları ve sipariş durumları config'den — bu bilgileri değiştirmek için kod açmana gerek yok. Admin gate check: is the sender in settings.admin_ids? If not, rejected. CSV at leads_csv_path is read and sent as a file via Telegram. Content: name, email, budget, timestamp. FAQ answers and order statuses come from config — no need to open code to update them.
